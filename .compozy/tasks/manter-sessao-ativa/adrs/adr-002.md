# ADR-002: Server-Side Authentication Wrapper and Cookie Lifespan Centralization

## Status
Accepted

## Date
2026-06-18

## Context
Following the product decision to extend user sessions to 30 days and provide seamless transparent renewal (ADR-001), we need a secure, maintainable architectural implementation strategy. Session lifetime literals should not be scattered. Moreover, validating authentication, athlete session existence, and potential token expiration across multiple Next.js API routes needs a centralized pattern to avoid code duplication, prevent infinite client-side redirect loops, and conform to the project's security standards.

## Decision
1. **Central Constants Configuration:** Define and export `SESSION_MAX_AGE` (equal to 30 days in seconds: `30 * 24 * 60 * 60`) within `src/config/index.ts`.
2. **Server-Side API Route Middleware Wrapper:** Implement a Higher-Order Function / wrapper `withProtectedAPI(handler)` in a server-focused security layer. This wrapper will:
   - Perform session cookie retrieval and validate via `getAuthenticatedAthleteId(req)`.
   - Return a `401 Unauthorized` response with a structured JSON signature if the cookie or session is missing or invalid.
3. **Server-Side Page Redirects:** In server-rendered page entries (such as `src/pages/index.tsx` inside `getServerSideProps`), check for active cookies. If missing or invalid, immediately leverage Next.js native `redirect` object targeting `/api/oauth/start` to execute a lightweight, fast, server-controlled transparent renewal sequence.

## Alternatives Considered
* **Client-Side Axios Interceptor Redirects:** Inspecting 401s in client-side Axios filters and mutating `window.location.href`. Reverted because multiple parallel asynchronous queries (SWR polls) could invoke overlapping, race-condition heavy redirects, risking browser navigation stuttering or cascading page flashes.
* **Inline Route Validation:** Copy-pasting auth and exception checks inside every single route. Rejected due to clear violation of DRY (Don't Repeat Yourself) guidelines and higher future maintenance overhead.

## Consequences
* **Positive:** Secure, single-responsibility authorization layer. Extremely clean Next.js pages and API route endpoints since auth logic is isolated. Prevents unnecessary client-side rendering cycles when an outdated session is detected.
* **Negative:** Technical developers must remember to wrap new protected API routes with `withProtectedAPI` instead of making standard handlers.
